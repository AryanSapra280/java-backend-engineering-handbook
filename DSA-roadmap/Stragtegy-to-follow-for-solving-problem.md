# DSA Problem Solving Strategy

> **Goal:** Crack SDE-2 interviews at Google, Microsoft, Amazon, Atlassian, Uber, Rubrik, Walmart, Adobe and other product-based companies.

---

## Online Assessment Mindset

Most first-round coding assessments have:

* **2 Questions**
* **90 Minutes**
* Usually **1 Easy/Medium + 1 Medium/Hard** or **2 Mediums**

Target:

* Finish Q1 in **20-30 minutes**
* Spend **55-65 minutes** on Q2
* Keep **5-10 minutes** for testing and edge cases

---

# Problem Solving Workflow

## Step 1: Understand the Problem (5-7 min)

Do **NOT** start coding immediately.

Ask yourself:

* What is the input?
* What is the output?
* What are the constraints?
* Can I solve it using brute force?
* What edge cases exist?

Draw an example if needed.

---

## Step 2: Identify the Pattern (5-10 min)

Before coding, think about which pattern fits.

Common patterns:

* Arrays
* HashMap
* Two Pointers
* Sliding Window
* Prefix Sum
* Binary Search
* Stack
* Queue
* Heap
* Greedy
* Backtracking
* Graph
* Dynamic Programming

If a pattern isn't obvious, try writing the brute-force solution first.

---

## Step 3: Think About the Optimal Solution (15-20 min)

Questions to ask:

* Can I reduce time complexity?
* Can I reduce space complexity?
* Can I preprocess data?
* Can I use sorting?
* Is there a known algorithm or data structure that fits?

Avoid jumping into implementation before the approach is clear.

---

## Step 4: Implement (20-30 min)

While coding:

* Write clean code
* Use meaningful variable names
* Handle edge cases
* Dry run with sample input

---

## Step 5: Debug (5-10 min)

Before submitting:

* Empty input
* Single element
* Duplicate values
* Negative values (if applicable)
* Overflow cases
* Large inputs

---

# When to Look at the Solution

## Medium Problems

### If after 25-30 minutes:

* You have **no direction**
* You cannot identify the pattern

👉 Read **only the intuition or approach**.

Do **NOT** read the implementation.

Close the solution and implement it yourself.

---

### If you know the algorithm but are stuck while coding

Spend another **10-15 minutes** debugging.

If still blocked, check only the specific implementation detail.

---

## Hard Problems

Hard problems often introduce a new algorithm or pattern.

If after **35-40 minutes** you still don't know the direction:

* Read the intuition.
* Understand the algorithm.
* Close the solution.
* Implement it from scratch.

---

# Never Do This

❌ Read solution

❌ Copy code

❌ Submit

❌ Move to next problem

This builds recognition, not problem-solving ability.

# After Every Problem

Record the following:

* Pattern
* Time Complexity
* Space Complexity
* Key Insight
* Mistakes Made
* Similar Problems

---

# Revision Strategy

### First Revision

Within **2 days**

---

### Second Revision

After **1 week**

---

### Third Revision

After **1 month**

If you can solve it without hints after one month, the pattern is likely internalized.

---

# Daily Goal

* 2 New Problems
* 1 Revision Problem
* Review mistakes
* Push clean code to GitHub

Consistency beats volume.

---

# Golden Rule

> Don't measure progress by the number of problems solved.
>
> Measure it by the number of patterns you can recognize and implement confidently under interview time constraints.
